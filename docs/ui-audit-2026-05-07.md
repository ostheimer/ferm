# UI-Audit 2026-05-07

## Zweck

Festgehaltener Befund-Stand der Web- und Mobile-Oberfläche von hege zum Stichtag **2026-05-07**, erstellt gegen Production (`https://hege.app`) und gegen den aktuellen Branch-Stand. Dieses Dokument ist eine Momentaufnahme: die Empfehlungen werden nicht hier abgehakt, sondern in [TODO.md](../TODO.md). Erledigte Punkte tragen unten in der Tabelle ein Datum.

## Methodik

- Production-Login als `ostheimer` (Rolle `ausgeher`) und als `revieradmin` (Rolle `revier-admin`), jeweils mit Demo-PIN `9526`.
- Manueller Walk-Through aller Backoffice-Routen unter `/app`.
- Code-Walkthrough für die Mobile-App; ein lokaler iOS-Simulator-Lauf war an einem CocoaPods-Encoding-Bug (`/opt/homebrew/Cellar/ruby/4.0.1`) und fehlenden `node_modules` im Worktree blockiert. Mobile-Befunde sind daher Code-basiert; Bestätigung per Geräte-Smoke folgt im nächsten Pass.
- Visuelle Evidenz (Screenshots) wird im selben Branch nachgereicht; Ablageziel ist `docs/assets/ui-audit-2026-05-07/`.

## Findings — Web-Backoffice

### F-01 Sidebar zeigt rollenfremde Links und redirected stillschweigend — *Kritisch*

**Reproduktion:** Login als `ausgeher` (Account `ostheimer`). Klick auf Sidebar-Link „Sitzungen" oder direkter Aufruf `/app/sitzungen`. Server antwortet mit `redirect("/app")` ohne Hinweis. Active-State der Sidebar zeigt weiter „Dashboard". User merkt nicht, dass der Klick nicht ausgeführt wurde.

**Code-Stellen:**
- Guard, der den Redirect auslöst: [apps/web/src/server/auth/guards.ts:32](../apps/web/src/server/auth/guards.ts#L32)
- Sitzungen-Route nutzt `requirePageRoles(["schriftfuehrer", "revier-admin"])`: [apps/web/src/app/app/sitzungen/page.tsx:8](../apps/web/src/app/app/sitzungen/page.tsx#L8)
- Sidebar liefert für alle Rollen dieselbe Liste: [apps/web/src/components/shell.tsx:7-14](../apps/web/src/components/shell.tsx#L7-L14)

**Empfehlung:**
1. Sidebar-Items rollen-aware filtern (Allowlist pro Item).
2. Ein dedizierter `/app/forbidden`-Pfad oder Toast-Hinweis statt stillem Redirect, damit der Bug nicht weiter unsichtbar ist.

### F-02 Production-DB enthält großen E2E-Test-Datenbestand — *Kritisch*

**Beobachtung:** Als `revier-admin` zeigt `/app/sitzungen` 15 Einträge — alle mit Titel `E2E Freigabe 1775426…`. `/app/protokolle` zeigt 7 Einträge derselben Herkunft mit Beschluss „Freigabe für den PDF-Test vorbereiten". `/app/fallwild` zeigt echte Demo-Daten gemischt mit Vorgängen `E2E Gemeinde 1775426…/E2E-Landesstrasse`.

**Wirkung:** Wer immer die Production-Demo zu sehen bekommt (Käufer, Stakeholder), sieht primär Test-Müll. Der Effekt überstrahlt jede UI-Politur.

**Empfehlung:**
1. Sofortmaßnahme: alle Datensätze mit Titel-/Gemeinde-Prefix `E2E ` aus Production löschen.
2. Strukturmaßnahme: E2E-Suite gegen einen separaten Neon-Branch laufen lassen (Vercel Preview-Branch ist bereits separat; Production-Schreibtests gehören dorthin).
3. Falls strukturelle Trennung nicht sofort machbar: Cleanup-Hook am E2E-Run-Ende, der per Prefix oder Tag erstellte Daten zuverlässig entfernt.

### F-03 „Kartenlage / Revier im Tagesbetrieb" ist eine Fake-Map — *Kritisch*

**Beobachtung:** Der Block auf dem Backoffice-Dashboard zeigt ein dunkelgrünes Grid mit drei Markern, deren Position aus `lat % 1` und `lng % 1` als CSS-Position berechnet wird. Daneben Pill „Google Maps geplant". Für ein Reviermanagement-Produkt ist das der zentrale Anker — und gerade hier sieht es nach Mock aus.

**Code-Stelle:** [apps/web/src/components/territory-panel.tsx:9-32](../apps/web/src/components/territory-panel.tsx#L9-L32)

**Empfehlung:** Echte Karte auf Basis Google Maps JavaScript API (`@vis.gl/react-google-maps`), zentriert auf das aktive Revier (`zentrumLat/Lng/Label` ist bereits in `reviere`-Tabelle), Marker für aktive Ansitze, Reviereinrichtungen und neueste Fallwild-Vorgänge. `GOOGLE_MAPS_SERVER_API_KEY` ist serverseitig schon im Einsatz; Client-Key trennen und in Vercel Env eintragen.

### F-04 Sitzung-Detail bleibt im Status `freigegeben` editierbar — *Hoch*

**Beobachtung:** Sitzung-Detail einer als „freigegeben" markierten Sitzung zeigt Titel, Termin, Ort weiterhin als Eingabefelder mit „Stammdaten speichern"-Button. Erwartet: Lock + sichtbarer „Neue Version öffnen"-Pfad.

**Code-Stelle:** [apps/web/src/app/app/sitzungen/[id]/page.tsx](../apps/web/src/app/app/sitzungen/%5Bid%5D/page.tsx) und [apps/web/src/app/sitzungen/[id]/sitzung-detail-client.tsx](../apps/web/src/app/sitzungen/%5Bid%5D/sitzung-detail-client.tsx)

**Empfehlung:** `disabled`-Attribut auf den Stammdaten-Inputs, sobald `sitzung.status === "freigegeben"`. Save-Button durch CTA „Neue Version öffnen" ersetzen, der die bestehende `protokollVersionen`-Tabelle nutzt (Schema in [apps/web/src/server/db/schema.ts](../apps/web/src/server/db/schema.ts) bereits vorhanden).

### F-05 Detail-Link in der Sitzungen-Liste zeigt auf `/sitzungen/[id]` ohne `/app`-Prefix — *Hoch / sicherheitsverdächtig*

**Beobachtung:** Der „Detail öffnen"-Link in der Sitzungen-Karten-Liste verweist auf `/sitzungen/sitzung-…` (ohne `/app`-Prefix). Eine entsprechende Route existiert in [apps/web/src/app/sitzungen/](../apps/web/src/app/sitzungen/), also außerhalb des `/app`-Auth-Schirms. Bitte prüfen, ob die Route eine Rollen-/Auth-Guard hat oder ob Sitzungslinks ohne Login öffenbar sind.

**Empfehlung:** Audit der Route, dann entweder
1. Einheitlich nach `/app/sitzungen/[id]` umziehen (wie es Server-Page bereits ist), oder
2. Auf der `/sitzungen/[id]`-Route denselben Auth-Guard wie `/app/*` setzen.

### F-06 Rollen-Labels in CAPS und ohne Umlaute im Sitzung-Detail — *Mittel*

**Beobachtung:** Teilnehmer-Liste zeigt `AUSGEHER · AO-01`, `JAEGER · LH-07`, `SCHRIFTFUEHRER · MM-04`. Mapping-Funktion existiert bereits in [apps/web/src/components/shell.tsx:88](../apps/web/src/components/shell.tsx#L88) und liefert „Ausgeher", „Jäger", „Schriftführung" — wird im Detail aber nicht genutzt.

**Empfehlung:** `formatRoleLabel` aus `shell.tsx` in eine geteilte Helper-Datei ziehen (z. B. `apps/web/src/lib/roles.ts`) und an allen Anzeigestellen verwenden.

### F-07 Reviereinrichtungen-Status-Pill `wartung-faellig` — *Mittel*

**Beobachtung:** Status-Pill auf Reviereinrichtung-Karten zeigt den technischen Identifier `wartung-faellig` statt einer lesbaren Variante.

**Empfehlung:** Mapping `wartung-faellig` → „Wartung fällig", `gut` → „In Ordnung". Gleiches Verfahren wie F-06.

### F-08 Hero-Headlines zu groß und Engineer-Sprech — *Mittel*

**Beobachtung:**
- Backoffice-Dashboard-Hero `clamp(2.4rem, 4vw, 4.4rem)` läuft auf Desktop bei ~70 px. Drückt KPIs und Karte unter den Fold.
- Login-Hero (Public): „Backoffice und App jetzt mit echter Session." ist eine Sprint-Milestone-Botschaft, kein Nutzwert für Anmelder.
- Reviereinrichtungen-Hero spricht von „Server-Schicht" und „derselbe Datenbasis" — Implementierungsdetail.

**Code-Stellen:**
- Tokens: [apps/web/src/app/globals.css:249-254](../apps/web/src/app/globals.css#L249-L254)
- Login-Copy: [apps/web/src/components/login-form.tsx:67-69](../apps/web/src/components/login-form.tsx#L67-L69)
- Dashboard-Hero: [apps/web/src/app/app/page.tsx:21-26](../apps/web/src/app/app/page.tsx#L21-L26)

**Empfehlung:** Hero-Cap auf ca. `2.6rem` setzen. Auf Login: „Anmelden mit Benutzername und PIN" oder konkreter Nutzen. Auf Dashboard und Reviereinrichtungen: in der Mikrocopy auf den Rollennutzen statt auf Architektur fokussieren.

### F-09 Demo-Daten enthalten technische Identifier statt Personennamen — *Niedrig*

**Beobachtung:** Sidebar `Revierleitung Gänserndorf | Admin | RL-01`. „Revierleitung Gänserndorf" ist der Account-Name aus den Seed-Daten ([packages/domain/src/mock-data.ts:30](../packages/domain/src/mock-data.ts#L30)), nicht ein Personenname. Sieht im Demo-Kontext seltsam aus.

**Empfehlung:** Account `user-revierleitung` mit einem realistischen Demo-Personennamen befüllen (z. B. Anna Müller). Echte Käufer sehen ihren eigenen Namen, daher kein Blocker.

### F-10 Public Landing ohne Bildsprache — *Mittel*

**Beobachtung:** Außer dem Logo-Mark keine Bilder, Mockups oder Visualisierungen auf der Landing. Für ein „gut designtes" Produkt fehlt ein Hero-Visual (Backoffice-Mock mit Karte) und ein iPhone-Mock (Fallwild-Erfassung).

**Empfehlung:** Sobald F-03 umgesetzt ist, einen echten Backoffice-Screenshot als Hero-Bild verwenden. Kann als statisches Asset in `apps/web/public/landing/` liegen.

### F-11 Fallwild-Liste im Web ohne Erklärungs-Eyebrow — *Niedrig*

**Beobachtung:** Header-Aktionen sind „CSV-Export" und „Aktualisieren" — kein „Neuer Vorgang". Fachlich richtig (Erfassung läuft mobil), aber ohne Hinweis wirkt es nach fehlender Funktion.

**Empfehlung:** Eyebrow ergänzen: „Erfassung läuft über die hege-App. Hier nur Lese-, Filter- und Export-Sicht."

## Findings — Mobile-App (Code-basiert)

### F-12 6 Bottom-Tabs überschreiten Apple-HIG — *Hoch*

**Beobachtung:** Tab-Navigator definiert sechs Tabs: Heute, Ansitze, Einrichtungen, Fallwild, Meldungen, Protokolle. Apple HIG empfiehlt max. fünf. Auf iPhone SE truncieren Labels.

**Code-Stelle:** [apps/mobile/app/(tabs)/_layout.tsx:35-78](../apps/mobile/app/%28tabs%29/_layout.tsx#L35-L78)

**Empfehlung:** Reduktion auf 4 Tabs (Heute / Ansitze / Fallwild / Mehr) plus „Mehr"-Screen mit Liste der sekundären Destinationen (Reviereinrichtungen, Protokolle, Revierarbeit, Profil).

### F-13 Login-Wortmarke ist eine Bildmontage statt eines Logo-Assets — *Hoch*

**Beobachtung:** Im Login-Screen wird die Wortmarke „hege" aus einer separaten `<Image>` für das „h" und einem `<Text>ege</Text>` mit `marginLeft: -22` zusammengesetzt. Bricht bei Dynamic Type, Dark Mode, Screenshots.

**Code-Stelle:** [apps/mobile/app/login.tsx:135-138](../apps/mobile/app/login.tsx#L135-L138)

**Empfehlung:** Ein einzelnes SVG/PNG mit der vollständigen Wortmarke in `apps/mobile/assets/`.

### F-14 Mobile MapPreview ist hartkodiert — *Kritisch* (gleicher Geist wie F-03)

**Code-Stelle:** [apps/mobile/components/map-preview.tsx:11-15](../apps/mobile/components/map-preview.tsx#L11-L15)

**Empfehlung:** `react-native-maps` (Provider Google auf Android und iOS), zentriert auf Revier-Mittelpunkt, gleiche Marker-Logik wie Web.

### F-15 Mobile Dashboard hat zu viele Aktionen — *Mittel*

**Beobachtung:** Toolbar enthält Aktualisieren + Queue-Sync + Abmelden. Sechs Metric-Tiles, Karte, Queue-Liste mit Retry/Verwerfen-Buttons, „Eingeloggt"-Card, „Letzte Benachrichtigung"-Card. Für einen mobilen Heute-Screen zuviel.

**Code-Stelle:** [apps/mobile/app/(tabs)/index.tsx](../apps/mobile/app/%28tabs%29/index.tsx)

**Empfehlung:**
1. Logout in einen Profil-Tab oder ins „Mehr"-Menü.
2. Queue-Status auf einen Header-Indikator (z. B. Glocken- oder Cloud-Icon mit Badge) und Detail in einem Sheet.
3. Nur die wichtigsten 2–3 Metriken oben — Rest in den Detail-Tabs.

### F-16 Foto-Auswahl ist Library-first statt Kamera-first — *Mittel*

**Beobachtung:** Fallwild-Form öffnet `ImagePicker.launchImageLibraryAsync`, nicht die Kamera. Für Fallwild im Feld ist die natürliche Hauptaktion „Jetzt fotografieren".

**Code-Stelle:** [apps/mobile/app/(tabs)/fallwild.tsx:297](../apps/mobile/app/%28tabs%29/fallwild.tsx#L297)

**Empfehlung:** Primärbutton „Foto aufnehmen" → `launchCameraAsync`, sekundär „Aus Bibliothek wählen".

### F-17 Choice-Chips für Enums skalieren schlecht auf 375 px — *Niedrig*

**Beobachtung:** Wildart-Auswahl rendert 7 Chips als horizontale Wrap-Reihe. Auf iPhone SE wird das mehrzeilig.

**Empfehlung:** Action Sheet oder Picker für Wildart, Geschlecht, Altersklasse, Bergungsstatus.

### F-18 Strings sind „Engineer-Deutsch" — *Niedrig*

**Beobachtung:** „Queue", „Synchronisiert…", „Queue konnte nicht synchronisiert werden", „1 gespeicherte Stände".

**Empfehlung:** Mikrocopy-Pass mit jagdfremder Tester:in. Mindestens: „Queue" → „Warteschlange" oder „Offene Meldungen". „1 gespeicherte Stände" → „1 Version gespeichert".

### F-19 Kein Dark Mode — *Niedrig*

**Beobachtung:** `userInterfaceStyle: "light"` in [apps/mobile/app.json](../apps/mobile/app.json) erzwingt hellen Modus. Outdoor-Nutzung bei Dämmerung profitiert von dunklem Theme.

**Empfehlung:** Auf `automatic` umstellen, sobald Tokens existieren (siehe F-21).

## Findings — Cross-cutting

### F-20 Public Registration legt nur `revier-admin` an, kein Member-Invite — *Hoch*

**Beobachtung:** [apps/web/src/server/modules/public-registration/service.ts:71](../apps/web/src/server/modules/public-registration/service.ts#L71) hardcoded `role: "revier-admin"`. Kein Endpoint zum Einladen oder Anlegen weiterer Mitglieder existiert.

**Wirkung:** Andere Rollen (jaeger, schriftfuehrer, ausgeher) können nur per Seed-Skript oder Direkt-DB-Eingriff existieren. Demo der Mehr-Rollen-Realität ist nur über die vier Demo-Logins (siehe [test-accounts.md](./test-accounts.md)) machbar.

**Empfehlung:** `POST /api/v1/memberships/invite` als Endpoint, einfacher Invite-Flow per E-Mail-Link mit zeitlich begrenztem Token; Rollen-Vergabe durch Revier-Admin.

### F-21 Keine geteilten Design-Tokens zwischen Web und Mobile — *Hoch*

**Beobachtung:** Web nutzt CSS-Custom-Properties in [apps/web/src/app/globals.css:1-13](../apps/web/src/app/globals.css#L1-L13), Mobile nutzt `colors`-Konstanten in `apps/mobile/lib/theme.ts`. Werte driften (z. B. „accent" hat in CSS `#9db36f`, in Mobile mutmaßlich nicht denselben Hex).

**Empfehlung:** Ein neues Package `@hege/tokens` (oder Erweiterung von `@hege/domain`) mit Farben, Spacing, Radius, Typo. Web bezieht via JS-Import + CSS-Generierung; Mobile importiert direkt.

### F-22 Kein Icon-System auf Web — *Niedrig*

**Beobachtung:** Sidebar zeigt nur Text. Mobile nutzt `@expo/vector-icons` (Ionicons). Web hat keine Icons.

**Empfehlung:** Lucide-Subset oder eigene SVGs in `apps/web/src/components/icons/`.

## Tooling-Findings (Hygiene, kein UI)

### F-23 Lokaler iOS-Build scheitert an CocoaPods/Ruby-Encoding — *Mittel*

**Beobachtung:** `expo run:ios` führt `pod install` aus, das mit `Encoding::CompatibilityError` in `/opt/homebrew/Cellar/ruby/4.0.1/lib/ruby/4.0.0/unicode_normalize/normalize.rb:153` abbricht. Workaround `LANG=en_US.UTF-8` half nicht; sekundär: `cannot load such file -- ./scripts/autolinking` weil `apps/mobile/node_modules` im Worktree fehlen.

**Empfehlung:** CocoaPods-Version pinnen, EAS-Preview-Build dokumentieren, native Smoke primär über TestFlight statt lokaler Pod-Kette.
